**Step 1: Design the shared helper**

Create `global-chart.inheritedJobPodSpec` in `_helpers.tpl` at the end of the file.

The helper accepts a dict with these keys:
```yaml
{{- $ctx := dict
  "root" $root                    # Root context ($)
  "deployName" $deployName        # Deployment key name
  "deploy" $deploy                # Deployment values map
  "jobName" $name                 # Job/cronjob name
  "job" $command                  # Job values map (the hook command or cronjob)
  "jobFullname" $hookFullname     # Pre-computed full name of the job
  "deployFullname" $deployFullname # Pre-computed deployment fullname
  "hasDeployConfigMap" $hasDeployConfigMap  # bool
  "hasDeploySecret" $hasDeploySecret       # bool
  "includeDnsConfig" true/false   # CronJobs: true, Hooks: false
-}}
```

The helper renders the pod spec body from `imagePullSecrets` through `tolerations`, including:
1. imagePullSecrets resolution (job > deploy > global, using hasKey at each level)
2. hostAliases (job > deploy, using hasKey)
3. podSecurityContext (job > deploy, using hasKey)
4. serviceAccountName (already resolved by caller, passed as parameter -- OR resolve inside helper)
5. automountServiceAccountToken (job > deploy, using hasKey)
6. dnsConfig (controlled by `includeDnsConfig` flag -- job > deploy for cronjobs, standalone for hooks)
7. nodeSelector (job > deploy, using hasKey)
8. tolerations (job > deploy, using hasKey)
9. affinity (job > deploy, using hasKey)
10. Container section: image, imagePullPolicy, command, args, securityContext, env, envFrom, resources, volumeMounts
11. envFrom merging: deployment's generated CM/Secret + deployment's external refs + job's external refs
12. env/additionalEnvs concatenation: deployment's + job's

**What stays OUTSIDE the helper (in the calling templates):**
- SA resource creation (hooks need helm hook annotations on SA, cronjobs do not)
- SA name resolution (different annotation patterns between hooks and cronjobs)
- The CronJob `spec:` wrapper (schedule, concurrencyPolicy, historyLimits) and the `jobTemplate.spec.template.spec:` nesting
- The Hook `spec.template.spec:` nesting
- Volumes (these are rendered after the container section and vary)

**Actually, reconsider scope:** Given the complexity, the helper should focus on the CONTAINER spec portion and the pod-level fields that are duplicated. The SA resolution logic is already fixed (Plan 02) and has subtle differences between hooks and cronjobs. A pragmatic approach:

The helper renders the block starting from `imagePullSecrets:` through `tolerations:` (pod-level fields) AND the container spec (image, command, args, securityContext, env, envFrom, resources). The caller provides the pod-level indentation context via nindent.

**Step 2: Implement the helper**

- Scope: Deployment-level only; root-level hooks/cronjobs do NOT use this helper

Use `-}}` trim on conditional lines before literal content to avoid blank lines (per CLAUDE.md guidelines).
